<goal>
The repo has one reusable, test-backed source of truth for executable-vs-blocked
stage readiness, instead of scattered "ready" assumptions.
</goal>

<assumptions_constraints>
- Do not wire `rd-agent` or the four stage entrypoints in this plan; that
  belongs to Plan 02 after the core truth contract is stable.
- Runtime and dependency truth must come from repo-owned declarations, not
  hidden shell heuristics. Use `pyproject.toml` and `scripts/setup_env.sh`.
- Artifact and state truth must stay inside `StateStorePort`; do not add raw
  filesystem probes that bypass persisted snapshots.
- Unknown checks count as blocked. Do not add warnings-only fallback paths.
</assumptions_constraints>

<pseudocode>
1. Read `pyproject.toml`, `scripts/setup_env.sh`, `state_store.py`,
   `artifact_state_store.py`, `recovery_service.py`, and `resume_planner.py`
   to lock the exact runtime/dependency declarations and the current false-ready
   wording.
2. Create `tests/test_phase23_preflight_service.py` first.
   if the test file does not assert runtime, dependency, artifact, state, and
   recovery blockers separately:
     extend the file until each blocker class has one exact test
     update artifact: `tests/test_phase23_preflight_service.py`
   else:
     keep the scope focused on canonical preflight truth only.
3. Add `v3/contracts/preflight.py` with one typed result model that includes:
   `readiness`, `recommended_next_skill`, `primary_blocker_category`,
   `primary_blocker_reason`, `repair_action`, and category-grouped blockers.
   if any field can express "unknown but okay":
     stop and revise the model so unknown remains blocked.
   update artifact: `v3/contracts/preflight.py`
4. Add `v3/orchestration/preflight_service.py`.
   - Parse `requires-python` from `pyproject.toml`
   - Require `uv` because `scripts/setup_env.sh` already treats it as mandatory
   - Require `pydantic` always
   - Require `pytest` and `import-linter` when the checked stage is `verify`
   - Load run, branch, stage, artifacts, and persisted recovery truth from
     `StateStorePort`
   for each blocker category in `runtime`, `dependency`, `artifact`, `state`,
   and `recovery`:
     evaluate the category against repo declarations or persisted snapshots
     append category evidence to the result
   if run/branch ownership, current-stage alignment, latest-stage truth,
   artifact presence, or recovery truth disagrees:
     mark the affected category blocked
   if any category fails:
     rank one primary blocker
     attach one exact repair action
     return `readiness="blocked"`
   else:
     return `readiness="executable"`.
   update artifact: `v3/orchestration/preflight_service.py`
5. Update `v3/orchestration/resume_planner.py`.
   if any message still contains the literal phrase `ready to run`:
     replace it with wording that says the stage is prepared and requires
     canonical preflight truth before execution
   else:
     keep the message edits minimal and factual.
   update artifact: `v3/orchestration/resume_planner.py`
6. Run the focused Phase 23 service gate.
   if tests fail:
     identify whether the failure is in runtime/dependency detection,
     snapshot consistency, artifact truth, or recovery truth
     revise the service or contract without weakening the blocked-on-unknown
     rule
   else:
     record the passing command and stop.
</pseudocode>

<edge_cases>
- `pyproject.toml` is missing or lacks `requires-python`; this must block with
  a `runtime` reason instead of assuming compatibility.
- `uv` is missing even though Python version is acceptable.
- A stage snapshot lists artifact ids, but one or more underlying artifact
  snapshots cannot be listed from `StateStorePort`.
- Branch snapshot and latest stage snapshot disagree about status, iteration, or
  current stage ownership.
- A completed stage has reusable artifacts but no persisted recovery
  assessment.
</edge_cases>
