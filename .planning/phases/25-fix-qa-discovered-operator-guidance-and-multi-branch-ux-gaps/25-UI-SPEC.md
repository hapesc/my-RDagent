## Copywriting Contract

This is the core contract for Phase 25. Every operator-facing text element is specified below with exact patterns.

### Guidance Text Structure (3-part canonical shape)

All operator guidance follows the Phase 24 locked three-part shape. Phase 25 does not change this structure; it enforces completeness.

| Field | Pattern | Example |
|-------|---------|---------|
| `current_state` | `"Current state: {stage_label} for run {run_id} on branch {branch_id} {state_descriptor}."` | `"Current state: framing stage (framing) for run run-001 on branch branch-001 completed successfully."` |
| `routing_reason` | `"Reason: {why this path was chosen}."` | `"Reason: framing completed and prepared the build handoff."` |
| `exact_next_action` | `"Next action: {verb} {run_id} / {branch_id} with {skill}."` | `"Next action: continue run-001 / branch-001 with rd-code."` |

### Primary CTA Patterns (per surface)

| Surface | CTA Copy | Format |
|---------|----------|--------|
| Start new run (no paused) | `"Next action: start a new run with rd-agent in exploration mode. Confirm or modify the suggested branch hypotheses, then start."` | Always includes exploration mode recommendation |
| Start new run (paused exists) | `"Next action: continue the paused run first. If you want to start fresh instead, explicitly request a new run."` | Paused continuation takes priority |
| Stage completed | `"Next action: continue {run_id} / {branch_id} with {next_skill}."` | Always paired with `next_step_detail` skeleton |
| Stage blocked (preflight) | `"Next action: resolve the {blocker_category} blocker before continuing."` | Paired with `repair_action` |
| Stage reused | `"Next action: continue {run_id} / {branch_id} with {next_skill} â€” previous results are still valid."` | Always paired with `next_step_detail` skeleton |

### Multi-Branch Hypothesis Copy

When `route_user_intent` detects a research-type intent and no paused run exists:

| Element | Copy Pattern |
|---------|-------------|
| Hypothesis line format | `"  - {hypothesis_text}"` (2-space indent, dash, space) |
| Hypothesis count | Exactly 3 hypotheses generated |
| Hypothesis template | `"Approach A: primary method for {intent_summary}"`, `"Approach B: alternative method for {intent_summary}"`, `"Approach C: baseline comparison for {intent_summary}"` |
| Intent summary truncation | First 50 characters of stripped user intent |
| Routing reason with hypotheses | `"Reason: plain-language intent suggests a research task. I suggest exploring these directions:\n  - {h1}\n  - {h2}\n  - {h3}"` |
| User modification prompt | Included in `exact_next_action`: `"Confirm or modify the suggested branch hypotheses, then start."` |

### Copy-Pasteable Skeleton Format (next_step_detail)

Phase 25 mandates `next_step_detail` on EVERY guidance path. The `detail_hint` field is removed entirely.

| Skeleton Type | Format | Example |
|---------------|--------|---------|
| Start skeleton | `title="Standalone V3 task" task_summary="{intent}" scenario_label="research" stage_inputs.framing.summary="Summarize the first framing step." stage_inputs.framing.artifact_ids=["artifact-plan-001"]` | Used by `build_start_new_run_guidance` |
| Start skeleton (exploration) | Same as above + ` exploration_mode="exploration" branch_hypotheses=['Approach A: ...', 'Approach B: ...', 'Approach C: ...']` | Multi-branch start path |
| Continuation skeleton | `run_id="{run_id}" branch_id="{branch_id}" summary="Summarize the current step." artifact_ids=["artifact-001"]` | Used by all stage completion, reuse, replay, review paths |

**Constraint:** Skeletons must be single-line strings. No embedded newlines. The consumer copies and pastes the entire string as arguments.

### Empty State Copy

| Scenario | Copy |
|----------|------|
| No paused runs, no intent | `"Current state: no paused run is active in the current working context, so a new run can start (start_new_run)."` |
| No runs exist at all | Same as above â€” the system does not distinguish "never used" from "no active run" |

### Error / Blocker State Copy

| Scenario | Copy Pattern |
|----------|-------------|
| Preflight blocked (runtime) | `"Current state: {stage_label} for run {run_id} on branch {branch_id} is blocked by a runtime prerequisite."` + `current_blocker_reason` with specific missing dependency + `repair_action` with exact fix command |
| Preflight blocked (data) | Same pattern, `current_blocker_category` = `"data"`, `repair_action` describes which file/artifact is missing |
| Stage blocked (recovery) | `current_action_status` = `"blocked"`, `routing_reason` explains the blocker, `repair_action` provides fix guidance |

### Destructive Actions

Phase 25 has no destructive actions in the operator guidance surface. The `disposition` -> `recovery_assessment` rename is a code-level change, not an operator-facing destructive action.

### Materialized Next-Stage Copy

When a stage completes and the next-stage NOT_STARTED snapshot is auto-created:

| Field | Value |
|-------|-------|
| `summary` | `"Prepared and requires preflight before execution."` |
| `status` | `NOT_STARTED` |
| `stage_iteration` | `1` |
| `artifact_ids` | `[]` (empty list) |

## Structured Output Shape Contract

This section defines the `structuredContent` JSON shape that programmatic consumers depend on.

### Outcome Field (SC-5)

All 4 stage entries must include `"outcome"` at the top level of `structuredContent`:

```
{
  "structuredContent": {
    "owned_stage": "<stage_key>",
    "outcome": "<outcome_value>",        // MANDATORY on every path
    "operator_guidance": { ... },
    "decision": { ... },
    "run": { ... },
    "branch_before": { ... },
    "branch_after": { ... },
    "stage_after": { ... }
  }
}
```

Valid `outcome` values (exhaustive):

| Value | Meaning |
|-------|---------|
| `"completed"` | Stage executed successfully |
| `"blocked"` | Stage execution blocked (non-preflight) |
| `"preflight_blocked"` | Preflight check failed before execution |
| `"reused"` | Previous results reused, no re-execution |
| `"replay"` | Stage replayed from existing artifacts |
| `"review"` | Stage requires operator review |

### Recovery Assessment Field (SC-6)

The `decision` object uses `recovery_assessment` (not `disposition`):

```
{
  "decision": {
    "recovery_assessment": "<RecoveryDisposition_value>",
    // ... other decision fields
  }
}
```

### Operator Guidance Payload Shape

The `operator_guidance` object within `structuredContent` follows the `OperatorGuidance` model with `detail_hint` removed:

```
{
  "operator_guidance": {
    "current_state": "<string, min_length=1>",
    "routing_reason": "<string, min_length=1>",
    "exact_next_action": "<string, min_length=1>",
    "recommended_next_skill": "<string, min_length=1>",
    "current_action_status": "<string | null>",
    "current_blocker_category": "<string | null>",
    "current_blocker_reason": "<string | null>",
    "repair_action": "<string | null>",
    "next_step_detail": "<string | null>"
  }
}
```

**Removed field:** `detail_hint` is deleted from the `OperatorGuidance` Pydantic model. Any caller passing `detail_hint` will get a Pydantic `ValidationError` due to `extra="forbid"`.

**Completeness rule:** `next_step_detail` must be non-null on every path except preflight-blocked paths where repair is the immediate action.

### RunStartRequest Schema Extension (SC-2)

```
{
  "exploration_mode": "exploration",     // default, or "convergence"
  "branch_hypotheses": [                 // optional, list of strings
    "Approach A: ...",
    "Approach B: ...",
    "Approach C: ..."
  ]
}
```
